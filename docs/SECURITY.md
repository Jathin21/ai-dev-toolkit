# Security

This app does two things that, done carelessly, would be dangerous:

1. Executes SQL generated by a language model against your database.
2. Sends code from private repositories to a third-party API (OpenAI).

This document explains the threat model for each and the mitigations in place. If you're deploying this in production — especially against a database that contains anything more sensitive than developer metadata — **read this carefully**.

## Threat model: NL-to-SQL

### Assumption: the LLM is untrusted input

We treat every SQL string the model produces as if an unauthenticated internet user typed it. This matters because:

- **Prompt injection is real.** A malicious PR description or commit message, once embedded into a prompt context, could coerce the model to ignore its instructions.
- **Models make mistakes.** Even without adversarial input, a model can produce `DELETE FROM users` when asked `"delete my cached preferences"`.
- **Models are trained on corpora that contain SQL injection payloads.** They know how to generate them.

The safety story therefore cannot rest on prompting alone. It's defense in depth.

### Defense layer 1 — Curated schema

`DatabaseQuery::SchemaDescriber` presents only 5 tables to the model: `repositories`, `pull_requests`, `code_embeddings`, `queries`, `code_searches`. The model never sees the `users` table in the prompt, so it's unlikely to reference it at all.

This is *not* a security boundary by itself — the model could still guess that a `users` table exists — but it sharply reduces the rate at which problematic queries are even generated.

### Defense layer 2 — Static validation (`SqlValidator`)

Every generated SQL string passes through `DatabaseQuery::SqlValidator.validate!` before execution. The validator enforces, in order:

1. **Comment stripping first.** Both `-- line` and `/* block */` comments are removed *before* keyword scanning. This prevents `/* DROP TABLE users */ SELECT 1` from fooling the scanner. Running this step first is the critical ordering choice.

2. **Statement splitting with string-literal awareness.** We split on `;` but track whether we're inside `'single'` or `"double"` quotes so that `SELECT '; DROP TABLE x; --'` is one statement, not three. Anything that produces more than one statement is rejected.

3. **Must start with `SELECT` or `WITH`.** Every other statement type is rejected outright.

4. **Forbidden keyword scan.** Word-boundary regex against `INSERT|UPDATE|DELETE|TRUNCATE|DROP|ALTER|CREATE|GRANT|REVOKE|VACUUM|ANALYZE|REINDEX|CLUSTER|COPY|LOCK|CALL|EXECUTE|DO|DECLARE|SET|RESET|COMMIT|ROLLBACK|SAVEPOINT|RELEASE`. Any hit rejects the query.

5. **Table allow-list enforcement.** We scan for identifiers after `FROM` and `JOIN` and require them to be on the allow-list. Intentionally conservative: a subquery referencing a disallowed table via a CTE name is fine (it's in the allow-list), but directly naming `users` is not.

### Defense layer 3 — Execution sandbox

Even validated SQL runs under restrictions:

```sql
BEGIN;
SET LOCAL statement_timeout    = 5000;   -- 5 second hard cap
SET LOCAL transaction_read_only = on;    -- rejects writes at the DB level
-- execute validated query --
ROLLBACK;                                -- always, even on success
```

The transaction is always rolled back. Even if a bug in the validator allowed a mutation through, the ROLLBACK would undo it.

### Defense layer 4 — Dedicated read-only role (recommended in production)

The three defense layers above are enforced in application code. In production, you should **also** run the app against a Postgres role that has no write permissions on any table:

```sql
CREATE ROLE ai_readonly LOGIN PASSWORD 'xxxxx';
GRANT CONNECT ON DATABASE ai_dev_toolkit_production TO ai_readonly;
GRANT USAGE ON SCHEMA public TO ai_readonly;
GRANT SELECT ON repositories, pull_requests, code_embeddings, queries, code_searches TO ai_readonly;
-- note: NOT granted on users, or any table added later by default
```

Then set `DATABASE_QUERY_URL` to use this role, and configure `NaturalLanguageExecutor` to open its read-only connection against `DATABASE_QUERY_URL` instead of the primary. This is the fourth wall.

### What the validator does *not* catch

- **Expensive SELECTs.** A valid-but-slow query (`SELECT * FROM code_embeddings CROSS JOIN code_embeddings`) will still run until the 5s timeout fires. It won't corrupt data but it will use CPU. Mitigation: the timeout, plus rate limiting.
- **Data exfiltration via allowed tables.** If an allowed table contains sensitive data (e.g. if you added `github_token` to the `users` allow-list — don't), the validator won't stop it. Mitigation: keep the allow-list tight and the column list in `SchemaDescriber` curated.
- **Timing side-channels.** In principle an attacker who can submit NL queries and observe response times could probe row counts. Mitigation: queries are rate-limited and audited.

### If a new feature adds a table

**Read this before adding a migration that creates a new table:**

- If the new table must be queryable by NL: add it to `SqlValidator::ALLOWED_TABLES` *and* `SchemaDescriber::SCHEMA`, and explicitly decide which columns to expose.
- If it must not be queryable: do nothing. It's off-limits by default.

Do not be tempted to reflect on `ActiveRecord::Base.descendants` to auto-populate either list. The allow-list is deliberately hand-maintained so that expanding it is a conscious act that shows up in code review.

## Threat model: data sent to OpenAI

### What we send

- **Code chunks** (for embeddings) — entire source files from indexed repos, chunked into ~500-token pieces
- **PR diffs** (for summarization) — capped at 120KB, sent as part of the chat prompt
- **User-typed queries** — search strings and NL database questions

### What we don't send

- User email addresses, names, or any data from the `users` table
- GitHub tokens or any other secret
- Results of the NL-to-SQL execution (only the generated SQL and the explanation prompt which shows only the first 5 rows)

### Configuration

OpenAI's default API terms exclude API inputs/outputs from training. If you need stricter guarantees:

- Use an **Azure OpenAI** deployment — same API surface, enterprise-grade data handling
- Use a **zero-data-retention** OpenAI agreement (requires direct contract)
- Replace `AI::Client` with a self-hosted model — the client abstraction makes this a localized change

### Per-repository opt-out

A future enhancement (see roadmap) is a `send_to_ai` flag on `Repository` that allows users to track PRs and browse them in the UI without sending diffs to OpenAI. Implementation-wise this is straightforward: the summarizer checks the flag and no-ops.

## Authentication and authorization

- **Auth**: Devise with email/password, with bcrypt cost 12 in production.
- **AuthZ**: Pundit policies scope all resources to the owning user. The `admin` role gates the Sidekiq web UI and (in future) cross-user reporting.
- **Session cookies**: Signed + encrypted, `SameSite=Lax`, `Secure` in production.
- **CSRF**: Standard Rails protection on all non-API endpoints.
- **Rate limiting**: `Rack::Attack`, see [API.md](API.md).

## Secrets management

All secrets come from environment variables; none are ever committed to the repo. In production, use your platform's secret manager (Fly.io Secrets, Heroku Config Vars, AWS Parameter Store, etc.) to inject them.

The `config.filter_parameters` list (in `config/initializers/filter_parameter_logging.rb`) redacts password fields and all `*_token` / `*_key` fields from logs. Any new secret field should be added to that list.

## Reporting vulnerabilities

Please email `security@your-domain.example` rather than opening a public issue. We'll acknowledge within 48 hours.
